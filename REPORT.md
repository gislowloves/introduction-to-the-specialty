# Финальный отчёт по проекту Romashkovo Valley

## Описание системы

Romashkovo Valley — браузерная мини-игра по мотивам мира Смешариков.
Игрок управляет жизнью деревни: каждый день получает событие и выбирает
один из трёх вариантов решения. Выбор влияет на характеристики персонажей
и настроение деревни. Игровая сессия рассчитана на 7 дней.

Стек технологий: Python + Flask (сервер), C++ (игровая логика),
HTML/CSS/JS (интерфейс), Docker (контейнеризация).

## Архитектура

Проект разделён на три слоя:

- **Фронтенд** (`templates/index.html`, `static/`) — отображает события,
  кнопки выбора и статистику персонажей.
- **Сервер** (`app.py`) — Flask-приложение, принимает запросы от браузера,
  передаёт состояние игры в C++-бинарник и возвращает результат.
- **Игровая логика** (`camomile.cpp`) — компилируется в бинарник `game`,
  получает JSON на stdin, возвращает JSON на stdout.

Схема взаимодействия:

Браузер → Flask (app.py) → subprocess → game (C++) → JSON → Flask → Браузер

## Тестирование

Использован `pytest`. Написано 8 тестов двух типов.

**Unit-тесты** — проверяют функцию `make_initial_state` напрямую,
без запуска сервера:
- день равен 0
- настроение деревни равно 50
- все 9 персонажей присутствуют в stats
- все начальные характеристики равны 25

**Functional-тесты** — проверяют HTTP-маршруты через тестовый клиент Flask:
- `GET /` возвращает статус 200
- `POST /restart` сбрасывает состояние игры к начальному
- `GET /stats` возвращает корректный JSON после рестарта

Тесты интегрированы в CI и запускаются автоматически при каждом push.

```bash
pip install flask pytest
pytest tests/ -v
```

## Статический анализ безопасности (Semgrep)

Использован инструмент Semgrep (`semgrep scan --config=auto .`).
Было проверено 339 правил на 11 файлах.

Найдено 2 проблемы, обе исправлены:

- **Контейнер запускался от root** — добавлен `USER appuser` в Dockerfile.
- **Flask слушал на `0.0.0.0`** — убран параметр `host` из `app.run()`.

После исправлений повторный запуск показывает `0 findings`.

```bash
pip install semgrep
semgrep scan --config=auto .
```

## Композиционный анализ зависимостей (Trivy SCA)

Использован инструмент Trivy (`trivy fs .`).

При первом сканировании найдено 11 уязвимостей в двух библиотеках:

- **cryptography 3.4.8** — 10 уязвимостей, максимальный severity HIGH.
- **oauthlib 3.2.0** — 1 уязвимость, severity MEDIUM.

Обе библиотеки являются транзитивными зависимостями — в коде проекта
напрямую не используются, но были подтянуты автоматически другими пакетами.
После обновления до актуальных версий повторное сканирование показывает
`0 findings`.

SBOM (Software Bill of Materials) в формате CycloneDX сохранён
в файле `sbom.json`.

```bash
trivy fs . --format cyclonedx --output sbom.json
```

## CI

Настроен GitHub Actions (`.github/workflows/ci.yml`).
При каждом push и pull request в ветку `master` автоматически выполняется:

1. Установка зависимостей
2. Линтинг кода — `flake8`
3. Сборка C++-части — `g++`
4. Запуск тестов — `pytest`
5. Статический анализ — `semgrep`
6. Анализ зависимостей — `trivy`
